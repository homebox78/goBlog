### 핵심 진단
**"남의 글(경쟁 블로그) 학습 루프"는 돌고 있지만, "내 글 성과 루프"는 존재하지 않는다.**
발행하면 거기서 끝. 노출수·클릭률·체류시간·수익이 하나도 안 들어온다.
지금 대시보드 숫자(글 수·발행 성공·품질점수·AI 비용)는 **내부 생산 지표**지 독자 반응이 아니다.

## B. 성과 → 생성 반영 (자가발전 루프 닫기)
> 골격은 완성됐다. **표본이 모이면 스스로 켜진다** (성과 신호 6건부터 학습 시작).

- [x] B-1. 성과 상위/하위 글 비교 → 규칙 도출 → 생성 프롬프트 주입 (`self-learning.ts`)
  - 관측 지표: GSC 노출·클릭·평균순위 + **네이버 AI 인용 여부** + 제목 길이·본문 길이·H2 수·표·FAQ·발행 시각
  - ⚠️ **표본 6건 미만이면 학습하지 않는다** — 없는 데이터로 규칙을 지어내면 '배운 척'하는 시스템이 된다
  - ⚠️ 발행 직후 글(daysLive 작음)을 '실패'로 오독하지 않도록 프롬프트에 명시
  - 대시보드에 "관측 중 (0/6건)" 정직하게 표시
- [ ] B-2. 성과 하위 글 원인 진단 → 개선안 자동 제안 (승인 후 재작성)
- [ ] B-3. GSC **유입 쿼리**를 키워드 발굴에 역주입 (실제로 들어온 검색어 = 가장 확실한 신호)
- [ ] B-4. ⚠️ 단기 CTR만 좇아 자극적 제목으로 흐르지 않도록 가드 (사용자 방침)
- [x] B-5. **키워드 트렌드를 실제로 활용** (`trend-signal.ts`)
  - 모멘텀(순위 상승 + 연속 등장일 + 검색량 변화) → 키워드 종합점수에 ±10점 반영
  - 글 생성에 트렌드 맥락 주입: 신규 이슈면 **속보성**, 3일+ 지속이면 **깊이·완결성**으로 전략 분기
  - ⚠️ **발견: 시계열이 사실상 없었다.** 수집 때마다 상위 10위만 저장했는데 키워드가 회차마다 바뀌어
    **같은 키워드의 이력이 안 쌓였다** — 190행 전부 "오늘 처음 등장", 모멘텀 전부 0.
    저장 범위를 상위 30위로 넓혀 이력이 실제로 축적되게 수정. **내일부터 모멘텀이 살아난다.**

## D. 측정 신뢰성 계층 — **자가학습보다 먼저** (P0)
> 잘못 수집된 성과를 학습하면 데이터가 없는 것보다 나쁘다.
> "수집 성공"이 아니라 **"정확한 글에 정확한 성과가 연결됐는가"**를 검증해야 한다.

- [x] D-1. **URL 정규화 + 매칭 충돌 차단** (2026-07-14) — 쿼리 보존(WP `?p=`), 추적 파라미터 제거,
      같은 키에 여러 글이 걸리면 **매칭에서 제외**하고 경고. 실측: 발행 URL 157건 → 고유 키 145개, 충돌 0건
- [ ] D-2. **canonical/리다이렉트 최종 URL 저장** — GSC는 canonical 기준으로 집계한다.
      현재는 발행 시 받은 URL만 저장 → 리다이렉트·canonical이 다르면 미매칭
- [ ] D-3. **수집 상태를 3분류로 분리** — `데이터 0` / `수집 실패` / `보고 기준 미충족`.
      지금은 셋 다 "0"으로 보인다 (이미 partial: 미매칭 샘플·ambiguous 반환 중)
- [ ] D-4. **정합성 검사** — GSC 속성 전체 합계 vs 우리 DB 합계 비교. 클릭 > 노출 같은 불가능한 값 감지
- [ ] D-5. **데이터 상태 구분** — GSC 최신 3일치는 미확정(fresh)이다. **학습에는 확정(final) 데이터만 쓴다**
- [ ] D-6. `search_query_daily` 테이블 — 유입 쿼리를 글·날짜별로 보존 (B-3의 전제)

## F. 자가학습 신뢰도 — 표본 단계별 적용 강도 (P1)
> ⚠️ 현재는 **6건이면 곧바로 규칙을 프롬프트에 강하게 주입**한다. 이건 너무 이르다.

| 표본 | 지금 동작 | 바꿔야 할 동작 |
|---|---|---|
| 0~5건 | 관측만 | 관측만 (동일) |
| 6~19건 | **규칙 강제 주입** | 참고 인사이트로만 표시, 사용자 확인 후 적용 |
| 20~49건 | 〃 | 규칙 주입 + 신뢰도 표기 |
| 50건+ | 〃 | 자동 반영 |

- [ ] F-1. 표본 단계별 적용 강도 분리 (지금은 6건에서 바로 100% 반영 — 우연을 법칙으로 굳힐 위험)
- [ ] F-2. **비교군 정규화** — 발행 경과일·플랫폼·검색량대가 다른 글을 단순 비교하면 잘못 배운다
      (평균순위 2위인 글과 18위인 글의 CTR을 직접 비교해 "제목이 좋다"고 결론 내리면 안 된다)
- [ ] F-3. **학습 규칙 버전 관리** (`learning_rules` 테이블) — 근거 글 목록·신뢰도·적용 시점 저장, **롤백 가능**
- [ ] F-4. **생성 재현성** — 글마다 사용 모델·프롬프트 버전·학습 규칙 버전 저장.
      이게 없으면 성과가 좋아졌을 때 **왜 좋아졌는지 추적 불가**
- [ ] F-5. 다목적 성과 점수 — 단기 CTR만 좇으면 자극적 제목으로 흐른다 (B-4와 통합)

## goBlog 운영 원칙 (판단이 갈릴 때 이걸 따른다)

1. **"구현 완료"와 "운영 검증 완료"를 구분한다.**
2. **데이터가 없으면 학습하지 않는다.** 표본 없이 뽑은 규칙은 배운 척이다.
3. **수집 실패를 성과 0으로 기록하지 않는다.**
4. **성과가 좋아도 정확성·신뢰성을 해치는 규칙은 적용하지 않는다.**
5. 모든 자동 수정은 **원인·근거·버전·복구 경로**를 갖는다.
6. **플랫폼이 지원하지 않는 기능을 성공한 것처럼 표시하지 않는다.**
7. 단기 CTR보다 **장기 신뢰도·참여도·인용 가능성**을 우선한다.
8. 자동화는 발행량을 늘리는 수단이 아니라 **품질을 일정하게 유지하는 수단**이다.
9. 외부 API와 플랫폼은 **언제든 바뀔 수 있다**고 가정한다.
10. 완전 자동화에는 **비용 한도·비상 정지·백업·롤백**이 반드시 포함된다.

## 미착수 (우선순위 순 — 과설계 경계)

> 개인 운영 프로젝트다. **엔터프라이즈급 관측 인프라를 다 짓지 않는다.**
> 아래는 "언젠가"가 아니라 **막힐 때 꺼내 쓰는 목록**이다.

**P1 — 데이터가 들어오면 바로**
- G. 성과 하위 글 자동 진단 (노출 없음 / 노출은 있는데 순위 낮음 / 순위는 좋은데 CTR 낮음 → 원인별 처방)
- H. GSC 유입 쿼리 역주입 (= B-3) — 실제로 들어온 검색어가 가장 확실한 신호
- K. 오래된 글 자동 갱신 (경과일·순위 하락·출처 만료 트리거)

**P2 — 수익화 단계에서**
- M. AdSense 수집 + 광고 배치 정책 (본문 중간 삽입 금지, 참여율 하락 시 원복)
- E. GA4 연동 (체류시간·스크롤 깊이 — GSC엔 없는 지표)
- N. 플랫폼별 capabilities 테이블 (네이버는 GSC 대상 아님 등 — 지원 안 하는 걸 성공처럼 표시 금지)

**P3 — 남에게 줄 때만 필요**
- O. 글 상태 머신 + 작업 idempotency (지금은 재시작하면 이미지 단계가 날아감 → 30분 스윕으로 우회 중)
- P. 설치 마법사 / Q. 백업·복구·감사 / R. 알림 채널
- ⚠️ 이건 **1인 운영에는 과설계**다. 실제로 아플 때만 짓는다.

## 개발 시 주의 (실제로 사고 났던 것들)

- **이미지 생성은 배포된 서버에서만** — 로컬 개발 API로 만들면 `MEDIA_PUBLIC_URL`이 없어 죽은 링크가
  공유 운영 DB에 박힌다. (지금은 DB 가드가 막지만, 애초에 서버에서 하는 게 맞다)
- **이미지 재생성 후에는 새로고침** 하고 편집할 것. 오래된 화면에서 저장하면 본문이 되돌아간다
  (지금은 409로 막힘).
- **배포는 작업 중이 아닐 때** — 우아한 종료를 넣어 진행 중 요청은 살아남지만, 굳이 겹칠 필요 없다.

## 개발 중 얻은 교훈 (2026-07-14 추가)

- **403은 원인이 두 갈래다** — 'API 미사용 설정' vs '권한 없음'. 상태코드만 로그에 남기면 엉뚱한 곳을 뒤진다.
  오류에 **응답 본문을 함께 실어라**. (GSC 403 = Search Console API가 꺼져 있던 것)
- **GSC 데이터 시작점 = "속성 추가 시점"이지 "소유확인 시점"이 아니다.** 누군가 먼저 속성을 추가해 뒀다면
  소유확인 전 데이터도 있을 수 있다. 우리는 속성 자체가 오늘 처음 추가돼서 이전 데이터가 없는 것.
  → **속성 추가일과 소유확인일을 따로 기록해야** 나중에 "왜 데이터가 없지?"를 다시 안 헤맨다.
- **매칭 0건일 때 "성과 없음"과 "URL 안 맞음"은 다르다** — 미매칭 URL 샘플을 반환하게 해서 즉시 구분되게 함.
- **"데이터가 쌓이고 있다"와 "쓸 수 있는 데이터가 쌓이고 있다"는 다르다.** 트렌드 190행이 있었지만
  같은 키워드의 이력이 없어 기울기를 못 그렸다. **행 수가 아니라 "연결된 점의 수"를 봐야 한다.**
- **내부 링크는 플랫폼별로 달라야 한다.** 네이버 글에서 티스토리로 보내면 독자가 남의 집으로 샌다.
  대상 글이 그 플랫폼에 없으면 **링크를 지운다** — "그럴듯한 다른 URL"로 대신 걸면 조용한 오연결이다.
